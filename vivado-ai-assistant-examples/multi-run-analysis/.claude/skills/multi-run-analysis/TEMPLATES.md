## REPORT.md Structure (Insight-Focused)

The report is organized for maximum signal-to-noise. Instead of dumping raw data tables, it delivers insights with supporting data. Every section answers a specific question.

#### Section 1: Executive Dashboard

A quick-glance summary. No tables — just key numbers and a one-line verdict.

```markdown
# Multi-Run Implementation Analysis

## Executive Dashboard

**Device:** xcvp1802-lsvc4072-2MP-e-S (Versal Premium, 4-SLR SSI)
**Runs analyzed:** 18 completed, 0 incomplete
**Methodology violations:** 0 across all runs

### Closure Status
| | Run | WNS (ns) | TNS (ns) | Status |
|---|---------|----------|----------|--------|
| Winner | impl_16 | **-0.022** | -20.6 | 22 ps from closure |
| Runner-up | impl_1 | -0.098 | -160.5 | |
| Worst | impl_5 | -0.905 | -55,099 | TNS explosion |

**Verdict:** ForceSpreading.low placement (impl_16) achieves best result — 22 ps from timing closure. Single clock domain `clkout1_primitive_1` is the sole bottleneck with 2,096 failing endpoints.

**Utilization:** LUT 16.03–16.16%, BRAM 1619.5, URAM 64, DSP 24 — consistent across all runs, no concern.
```

If utilization varies significantly across runs (>2% LUT difference), add a brief note explaining why (different opt_design directives). Otherwise, one line is sufficient — do NOT include a full utilization table.

#### Section 2: Timing Progression Through PnR

Shows where timing is won and lost. This is the most actionable section.

```markdown
## Timing Progression Through PnR

| Run | Post-Place WNS | PhysOpt WNS | Post-Route WNS | PhysOpt Recovery | Route Degrad. | Iters |
|---------|----------------|-------------|----------------|------------------|---------------|-------|
| impl_16 | -1.062 | -0.045 | **-0.022** | 1.017 | +0.023 | 123 |
| impl_1 | -0.856 | -0.042 | -0.098 | 0.814 | -0.056 | 166 |
| impl_5 | -2.027 | -1.155 | -0.905 | 0.872 | +0.250 | 139 |

**Key insight:** impl_16 starts with the worst post-place WNS (-1.062) but phys_opt recovers 1.017 ns (best recovery). Routing adds only 0.023 ns — cleanest route degradation. impl_1 has better post-place WNS but loses 0.056 ns in routing.

**Phys_opt effectiveness:** All runs show strong phys_opt recovery (0.8–1.0 ns), confirming AggressiveExplore is appropriate. The bottleneck is post-place quality and route degradation, not phys_opt.
```

Include ALL runs in this table, sorted by post-route WNS (best first). Compute:
- **PhysOpt Recovery** = |Post-Place WNS| - |PhysOpt WNS| (positive = improvement)
- **Route Degradation** = Post-Route WNS - PhysOpt WNS (negative = routing worsened)

#### Section 3: Critical Path Pattern

Identifies WHAT is failing, not just how much.

```markdown
## Critical Path Pattern

### Failing Clock Domain
All 18 runs fail exclusively on **`clkout1_primitive_1`** (intra-clock setup):
- Winner impl_16: WNS = -0.022 ns, **2,096 failing endpoints** out of 881,530 total
- No inter-clock timing failures across any run

This is a single-clock-domain problem — the same ~2K endpoints fail regardless of strategy. Targeted optimization (constraint tuning, pipeline insertion) on this clock domain will be more effective than further directive sweeps.

### Clock Domain Summary (impl_16)
| Clock | WNS (ns) | Failing EP | Total EP | Status |
|-------|----------|-----------|----------|--------|
| clkout1_primitive_1 | **-0.022** | 2,096 | 881,530 | FAIL |
| clkout2_primitive_1 | 0.011 | 0 | 398,270 | PASS |
| clkout2_primitive | 0.002 | 0 | 169,421 | PASS |
| clk_pl_0 | 0.343 | 0 | 55,243 | PASS |
| All other clocks | — | 0 | — | PASS |
```

Include the top 5 clocks by endpoint count, plus any clocks with WNS < 0. If multiple clocks fail, note which ones are consistent across runs vs. strategy-dependent.

#### Section 4: Strategy Impact Summary

Groups runs by strategy pattern instead of listing 18 identical rows.

```markdown
## Strategy Impact

| Strategy Pattern | Runs | WNS Range | Post-Place WNS | Verdict |
|-----------------------------------------------|------|-----------|----------------|---------|
| ForceSpreading.low + ReduceCong/PinDens.low | impl_16 | -0.022 | -1.062 | **Winner** |
| Default placement (no directive) | impl_1,12,18 | -0.098 | -0.856 | Solid baseline |
| Explore placement (basic) | impl_4,9,10,13,17 | -0.358 | -1.044 | Identical results — wasted compute |
| AggressiveExplore placement | impl_3 | -0.342 | — | |
| Explore + ExtraTimingOpt.high | impl_11 | -0.419 | — | No benefit |
| Explore + EarlyBlockPlacement | impl_7 | -0.330 | — | |
| Explore + net_delay_weight high | impl_8 | -0.323 | — | |
| ForceSpreading.med + ReduceCong/PinDens.med | impl_15 | -0.385 | — | |
| ForceSpreading.high + ReduceCong/PinDens.high | impl_14 | -0.502 | — | **Avoid** |
| ExploreWithRemap | impl_5 | -0.905 | -2.027 | **Avoid — TNS explosion** |
| Explore + WLDrivenBlockPlacement | impl_6 | -0.360 | — | Only hold failure |
| BalancedSLR variants | impl_13,17,18 | -0.098 to -0.358 | — | |

**Key findings:**
- ForceSpreading sweep: `.low` >> `.med` >> `.high` — lower intensity preserves timing
- impl_4/9/10/13/17 produce identical timing — same seed, wasted compute
- ExploreWithRemap causes TNS explosion — do NOT use for this design
```

Group runs that share the same strategy pattern AND produce identical timing on one row. Note which strategies produced unique value vs. redundant results.

Include one device map for the **winner run** and one for the **worst run**. Then add a per-SLR comparison table for the winner, runner-up, and worst:

```markdown
### Per-SLR Hotspot Comparison

| SLR | impl_16 (winner) | impl_1 (runner-up) | impl_14 (worst cong.) |
|------|------------------|--------------------|-----------------------|
| SLR0 | E=1.19 (GT) | E=1.19 (GT) | E=1.19 (GT) |

- SLR0 EAST=1.19 is constant — hardened DCMAC/GT IP, not addressable
- SLR3 SOUTH is a recurring hotspot (1.11–1.21 across strategies) — design-level issue
```

Show actual per-SLR MaxCong values in the device map. Mark directions >1.1 with arrows and ▲/▼/◄/► indicators. Use `HOT`, `FIXED`, `clean` labels.

> **Note:** Relating congestion hotspots to RTL hierarchies requires `report_design_analysis -congestion` with a loaded DCP — see the `place-design-congestion-analysis` skill.

#### Section 6: Anomalies & Critical Observations

Only include findings that require action. Skip "no anomalies detected" if there are none — just omit the section header.

```markdown
## Critical Observations

1. **impl_6: Post-route hold failure** — WHS=-0.005 ns from WLDrivenBlockPlacement. Avoid this sub-directive.
2. **impl_5: TNS explosion** — -55,099 ns (44× worse). ExploreWithRemap caused excessive register duplication (971K FFs vs 961K baseline) without benefit.
3. **5 redundant runs** — impl_4/9/10/13/17 share identical timing. Replace with varied seeds.
4. **impl_4 PRPO marginal** — post-route phys_opt gained only 6 ps WNS improvement. Low ROI for the runtime.
```

#### Section 7: Recommended Next Steps

Numbered priority list tied to findings. Be specific — name the run, the clock domain, the directive.

```markdown
## Recommended Next Steps

1. **Close the 22 ps gap on impl_16:**
   - Run a third `phys_opt_design -directive AggressiveExplore` pass on the routed DCP
   - Or try `phys_opt_design -directive RewireToReduceTimingViolations`
   - Target clock domain: `clkout1_primitive_1`

2. **Investigate SLR3 South congestion** (recurring 1.11–1.21):
   - Load impl_16 DCP → `report_design_analysis -congestion` to identify RTL hierarchies
   - Use `place-design-congestion-analysis` skill

3. **Eliminate redundant runs:**
   - Replace impl_4/9/10/13/17 (identical results) with varied placement seeds or new strategies
   - Consider `ForceSpreading.low` + `BalancedSLR.low` combination

4. **Avoid in future sweeps:**
   - ExploreWithRemap (TNS explosion)
   - ForceSpreading.high (counterproductive wirelength)
   - WLDrivenBlockPlacement (hold violations)
```

Populate ALL runs in the `runs` array. Per-SLR congestion is only required for runs that have per-SLR data extracted (typically winner, runner-up, worst, and runs with global congestion >0.95). For other runs, `per_slr` can be an empty array `[]`.

The `strategy_groups` array groups runs by shared strategy pattern — this powers the Strategy Impact visualization.

The `critical_path_pattern` object summarizes the cross-run clock domain analysis.

Ensure all numeric fields use numbers (not strings). Use `null` for missing data instead of empty strings.

The `anomalies` array contains all critical observations from Step 8. Each entry has `severity` (`critical`, `warning`, or `info`) and `text`. The dashboard renders these with color-coded left borders.

The `next_steps` array contains the prioritized recommendations from the REPORT.md. Each entry has `priority` (integer, 1=highest) and `text`. The dashboard renders these as numbered action items.

The `tcl_commands` object per run stores the exact TCL command strings for each implementation phase. The dashboard renders these in a detail panel when clicking on any run row. Include `post_route_phys_opt` only if a post-route phys_opt pass was run (null otherwise).

The `common_patterns` object groups patterns into `adopt` (proven effective), `avoid` (proven harmful), and `observations` (insights). Each pattern has `pattern` (the TCL command or strategy), `frequency` (how many runs used it), `evidence` (measured impact), `verdict` or `recommendation`, and `phase` (which implementation phase).

### `detailed_congestion` schema (per run, OPTIONAL — Step 6b)

This field is populated ONLY when `report_design_analysis` output files are discovered from the run log. When absent, set to `null`. The skill discovers files by parsing `INFO: [Vivado 12-24828] Executing command : report_design_analysis` lines in runme.log, extracting the `-file` argument.

```json
"detailed_congestion": {
  "source_commands": [
    {
      "flags": "-congestion",
      "file": "impl_1/post_place_congestion.rpt",
      "type": "congestion"
    },
    {
      "flags": "-complexity",
      "file": "impl_1/complexity.rpt",
      "type": "complexity"
    }
  ],
  "rent_exponent": 0.65,
  "per_clock_region": [
    {"region": "X0Y0", "north": 1, "south": 2, "east": 1, "west": 0},
    {"region": "X1Y0", "north": 0, "south": 1, "east": 1, "west": 1}
  ],
  "top_congested_modules": [
    {"module": "u_core/u_datapath", "contribution_pct": 23.5},
    {"module": "u_core/u_ctrl", "contribution_pct": 12.1}
  ],
  "complexity": {
    "avg_logic_levels": 8.3,
    "max_logic_levels": 24,
    "logic_level_distribution": {"1": 450, "2": 1200, "3": 890, "4": 320}
  }
}
```

Fields within `detailed_congestion`:
- `source_commands` — array of discovered commands from the log, each with `flags`, `file` (resolved path), and `type` (`congestion`, `complexity`, or `qor_summary`)
- `rent_exponent` — from `-congestion` report; `null` if not found
- `per_clock_region` — from `-congestion` report; array of per-region N/S/E/W congestion levels
- `top_congested_modules` — from `-congestion` report; array of top modules contributing to congestion
- `complexity` — from `-complexity` report; `null` if no complexity report found

The dashboard conditionally renders a "Detailed Congestion" card in the Congestion tab when any run has `detailed_congestion` data.
